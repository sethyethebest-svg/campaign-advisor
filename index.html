Where it gets used

The use case is simple: weather changes delivery demand and promotion efficiency.

Each day, a growth or delivery-channel manager can decide which online platform campaign to run before budget is wasted.

Rain day

Shift spend toward delivery-platform acquisition.

When rain changes ordering behavior, the plan can move budget toward delivery-platform first orders, retargeting, or platform-funded promotions.

Hot day

Use lighter platform incentives when delivery demand is already high.

On warm days, the product can recommend margin-aware delivery offers instead of over-discounting natural beverage demand.

Platform reality

Show two online campaign options instead of one black-box answer.

The system ranks delivery-platform campaign types, then lets the operator approve the final choice before launch.

Why it is credible

The demo uses fixed rules today; production learns from live delivery data.

This web demo uses transparent sample coefficients so the logic is inspectable. In production, the same interface would update coefficients from live delivery-platform orders, campaign spend, conversions, payout, and local weather.

Data pipeline

  1. Weather forecast Browser location, BigDataCloud city lookup, and Open-Meteo 7-day forecast.
  2. Delivery outcomes Platform orders, campaign spend, conversion, payout, P-ROAS, and CAC.
  3. Live learning loop Demo uses fixed coefficients; production recalibrates them as live data changes.
  4. Operator decision Top 2 online delivery campaigns, metric toggle, and human approval.

Explainability

Fixed demo coefficients can become live-updated production coefficients

One additive model scores every day x campaign pair. Demo coefficients are fixed; production coefficients can refresh from live delivery-platform performance.

Scoring model Y_metric(day, campaign) = beta_0 + beta_campaign + beta_temp * (avg_temp_F - 68°F) + beta_rain * rain% + beta_wind * wind_mph
